PalApi — plugin API reference

framework 0.2.1 (beta)ABI v12

Everything a PalApi plugin can do, grouped by area. For each function: what it does, when a plugin uses it, and which thread it must run on. This complements PLUGIN_API.md; the whole API lives in one header, include/palapi/palapi.h.

What is a plugin? A native .dll. When the server boots, PalApi loads it and hands it a struct of function pointers (the PalApi struct). A plugin has two entry points:
extern "C" __declspec(dllexport) void Plugin_Init(const PalApi* api); // called on load extern "C" __declspec(dllexport) void Plugin_Unload(void); // called on unload (optional)
In Plugin_Init the plugin receives api and calls back through it (e.g. api->log.info("hello")). The ABI is append-only: check api->abi_version before using a field newer than the version you built against.
The two threads — read this first. Some functions must run on the game thread: anything that READS, WRITES or CALLS the game engine. You get there via run_on_game_thread, or from a callback that already runs there (chat, timer, ProcessEvent observer). Functions marked any thread are safe anywhere. Golden rule for every callback: return fast, never block — the game thread serves every player.
The plugin handle. Each plugin gets a unique id (api->plugin_handle). Pass it to anything you register (hooks, commands, timers): PalApi then removes all of your registrations automatically when your plugin unloads. No leaks, no crashes.
Trust model. A plugin is trusted native code, free to do anything the host can — including crashing the server on purpose (see host.panic). The security boundary is the operator's choice of which plugins to enable, not the framework second-guessing you. What the framework guarantees is attribution: when a plugin faults it is isolated and the log names the plugin (and the engine function for an observer); on a full crash a palapi_crash_*.txt names the last plugin that was running.

Logging — api>log

log.info / log.warn / log.error

void info(const char* message); void warn(const char* message); void error(const char* message);

Write a line to the server log (info / warning / error).

any thread
Used for: tracing what the plugin does; the first reflex for debugging.

Finding objects and classes — api>reflect / reflect2

PalApi resolves game classes and objects by name (no hardcoded addresses), which makes it robust across game updates.

reflect.find_class

uint64_t find_class(const char* class_name);

Resolve a game class definition by name. Returns its address, or 0 if absent.

any thread
Used for: checking a class exists on this build, or getting a starting point before reading/calling.

reflect.name_of / reflect.class_name_of

int name_of(uint64_t object, char* out, int cap); int class_name_of(uint64_t object, char* out, int cap);

Give an object's name, or its class name. Returns the length written (0 on failure).

any thread
Used for: knowing what you are working with, showing readable info.

reflect2.find_object_by_name

uint64_t find_object_by_name(const char* name);

The first object named exactly this (e.g. a class-default object). 0 if none.

any thread
Used for: reaching a known, unique game object.

reflect2.find_live_instance

uint64_t find_live_instance(const char* class_substring);

The first LIVE instance whose class name contains this text. 0 if none.

any thread
Used for: grabbing a real in-game object (e.g. the game state) to then read from or call.

reflect2.process_event_addr

uint64_t process_event_addr(void);

The verified address of ProcessEvent (the engine's universal call dispatch). 0 if not located.

any thread
Used for: advanced use. Normally you do NOT touch it directly: to react to ProcessEvent, use observers.

Calling engine functions — api>call (v2)

call.call_getter_int

int call_getter_int(const char* class_substring, const char* func_name, int* out);

Call a no-argument int getter on a live instance of the class. Returns 1 and fills *out on success.

game thread required
Used for: reading a value the game computes (e.g. max player count) through the real engine function.

call.call_function_ints

int call_function_ints(uint64_t instance, const char* func_name, const int* args, int argc, int* out_ret);

Call a function taking argc ints and returning an int, on the instance. 1 on success.

game thread required
Used for: triggering a game action that takes ints. (For other parameter types, see call2.)

Reading and writing state — api>state (v2)

state.read_int32 / state.read_float

int read_int32(uint64_t instance, const char* prop, int* out); int read_float(uint64_t instance, const char* prop, float* out);

Read a property (int or float) by name from an instance. 1 on success.

any thread
Used for: reading a precise value of an object (health, score, etc.).

state.resolve_field_addr

int resolve_field_addr(const char* class_substring, const char* prop, uint64_t* out_addr);

Give the exact address of a property on a live instance of a class. 1 on success.

any thread
Used for: getting a field address before writing it with write_scalar.

state.write_scalar

int write_scalar(uint64_t addr, const void* src, uint32_t n); // n in {1,2,4,8}

Write a small value (1..8 bytes) at an address, GUARDED: PalApi checks the region is writable and records the old value so the write can be reverted. 1 if accepted, 0 if the safety gate refuses. Never faults the server.

game thread required
Used for: changing a game value (e.g. raising a limit) safely, with an undo safety net.

state.undo_last

int undo_last(void);

Revert the most recent guarded write. 1 if one was reverted.

game thread required
Used for: rolling back a write_scalar.

Marshaling to the game thread — api>thread (v2)

thread.run_on_game_thread

int run_on_game_thread(void (*fn)(void* user), void* user, const void* plugin_handle);

Run fn(user) on the game thread (at the next tick). 1 if queued. Pass api->plugin_handle: if the plugin unloads first, the job is dropped (calling into an unmapped DLL would crash).

any thread runs the job on the game thread
Used for: the bridge. From anywhere (e.g. a background thread), go through here to safely read/write/call the game.

Low-level hooks — api>hook (v2)

hook.set_hook

int set_hook(uint64_t target_addr, void* detour, void** out_original, const char* label, const void* plugin_handle);

Detour a function: calls now go through the plugin, which can call the original (given in *out_original). Removed automatically on unload.

advanced / powerful
Used for: intercepting a specific function. Note: to react to ProcessEvent (the common target) do NOT use this — use observers (stacking hooks on one address crashes the server).

hook.remove_my_hooks

int remove_my_hooks(const void* plugin_handle);

Remove every hook set by this plugin. Returns how many were removed.

any thread
Used for: cleaning up before unload (also done automatically).

Observing engine calls — api>events (v3)

The right way to react to game events: PalApi installs ONE real hook on ProcessEvent and fans out to every observer. No hook stacking, no conflict.

events.register_processevent

int register_processevent(PalProcessEventCallback pre, PalProcessEventCallback post, void* user, const void* plugin_handle); // callback: void cb(void* self, void* function, void* parms, void* user);

Register a pre and/or post observer for every engine call. The callback receives the target object, the function being called, and its parameter buffer. Returns a handle > 0, or 0 on failure.

game thread (callback runs there)
Used for: reacting to what happens in game (a player speaks, an action fires...). This is the brick chat commands are built on. Rule: the callback is on the hot path — it must be VERY fast.

events.unregister_processevent

int unregister_processevent(int handle);

Remove one observer by handle. 1 if removed.

any thread
Used for: stopping observation (also done automatically on unload).

Broadcast to players — api>server (v3)

server.broadcast_message

int broadcast_message(const char* utf8_text);

Show a message on-screen to every connected player. 1 on success.

game thread required
Used for: announcing something, replying to a chat command.

Chat commands — api>commands (v4) / commands2 (v8)

commands.register_command

int register_command(const char* prefix, PalCommandCallback callback, void* user, const void* plugin_handle); // callback: void cb(const char* text, void* user);

Register a command: when a player types a line whose leading token is prefix (e.g. !ping), the callback is called with the full text. Handle > 0, or 0 on failure.

game thread (callback runs there)
Used for: giving players commands. The callback may reply (broadcast) or read/call the game directly.

commands2.register_command_ex

int register_command_ex(const char* prefix, const char* description, PalCommandCallback callback, void* user, const void* plugin_handle);

Like register_command, plus a short help line listed by the built-in !help.

game thread (callback runs there)
Used for: having your command show in !help. Registering several prefixes on the same callback = aliases.

commands.unregister_command

int unregister_command(int handle);

Remove one command by handle. Blocks until any in-flight call finishes.

any thread
Used for: disabling a command (also done automatically on unload).

RCON commands — api>rcon (v10)

Reply to a custom command sent by the remote admin tool (RCON), not by chat.

rcon.register_command

int register_command(const char* prefix, PalRconHandler handler, void* user, const void* plugin_handle); // handler: int h(const char* command, char* reply_out, int reply_cap, void* user); // return 1 = handled (reply_out goes back to the RCON client), 0 = not handled

Register an RCON command: when an admin types a line starting with prefix, the handler writes a reply and returns 1. The reply goes back to whoever typed it. Collisions with the native admin commands are refused.

return fast, NEVER block
Used for: admin commands (e.g. a status dump). Note: on the current build this handler runs on the game thread — if it blocks it freezes the WHOLE server. Just produce a short reply quickly.

rcon.unregister_command

int unregister_command(int handle);

Remove an RCON command by handle. 1 if removed.

any thread
Used for: disabling an RCON command (also done automatically on unload).

Timers — api>timers (v5)

timers.schedule_after

int schedule_after(uint32_t delay_ms, PalTimerCallback callback, void* user, const void* plugin_handle);

Run callback ONCE, delay_ms ms from now, on the game thread. Handle > 0.

callback runs on the game thread
Used for: deferring an action (e.g. "in 10s, announce...").

timers.schedule_every

int schedule_every(uint32_t interval_ms, PalTimerCallback callback, void* user, const void* plugin_handle);

Run callback REPEATEDLY every interval_ms ms, on the game thread. Handle > 0.

callback runs on the game thread
Used for: a recurring task (e.g. autosave, periodic announce).

timers.cancel_timer

int cancel_timer(int handle);

Cancel a timer by handle. 1 if removed.

any thread (outside a callback)
Used for: stopping a timer (also done automatically on unload).

Plugin config — api>config (v6)

A plugin reads ITS OWN settings, declared as a flat Config object in its PluginInfo.json: "Config": { "greeting": "hi", "max_kills": 10, "announce": true }.

config.get_string / get_int / get_double / get_bool

int get_string(const void* plugin_handle, const char* key, const char* def, char* out, int cap); int get_int(const void* plugin_handle, const char* key, int def); double get_double(const void* plugin_handle, const char* key, double def); int get_bool(const void* plugin_handle, const char* key, int def); // 1/0

Read a setting by key. If the key is missing (or the wrong type), returns the default def. In-memory only.

any thread, even from Plugin_Init
Used for: being configurable without recompiling. The server operator edits PluginInfo.json, the plugin adapts.

Connected players — api>players (v7)

List connected players and read a field per player. Read-only.

Cost note: count and for_each walk EVERY live game object (hundreds of thousands under load). Call them from a background thread or an occasional timer, never from a game-thread callback (chat, observer).

players.count

int count(const char* class_substring); // e.g. "PalPlayerState"

Count live instances whose class name contains this text.

any thread (but costly)
Used for: knowing how many players are connected.

players.for_each

int for_each(const char* class_substring, PalPlayerVisitor visitor, void* user); // visitor: int v(uint64_t player, void* user); return non-zero to continue, 0 to stop

Visit each player; the visitor is called for each. Returns how many were visited. The player address is valid only for the duration of the visitor call.

any thread (but costly)
Used for: doing something per player (read a field, count by a criterion...).

players.read_int / players.read_string

int read_int(uint64_t player, const char* prop, int* out); int read_string(uint64_t player, const char* prop, char* out, int cap);

Read a property (int or text) of a given player object. Cheap (one object).

any thread
Used for: reading a player's info (name, score, side...) during a for_each.

Typed calls — api>call2 (v9)

call2.call_function_typed

int call_function_typed(uint64_t instance, const char* func_name, const PalApiValue* args, int argc, void* ret_buf, uint32_t ret_size); // PalApiValue = { uint32_t size; const void* data; } (one parameter = size bytes at data)

Call a function with mixed-type parameters (int, float, bool, or a plain-data struct like a vector) and read back a scalar/struct result. PalApi CHECKS each parameter's size against the live reflection and refuses on mismatch. Engine-managed types (FString, TArray...) are NOT allowed here — use dedicated functions (e.g. broadcast) for those.

game thread required
Used for: calling a richer game function than "just ints" (e.g. one that takes a position vector).

Deliberate crash, attributed — api>host (v11)

Trust model: a plugin is trusted native code, FREE to do anything, including shutting the server down on purpose. Security = the operator's choice of plugins. PalApi's job is not to prevent, but to say clearly who crashed and on which function.

host.panic

void panic(const void* plugin_handle, const char* reason); // does not return

Shut the server down DELIBERATELY, attributed. Logs PANIC: plugin '<name>': <reason>, writes a palapi_panic_*.txt next to the logs (plugin name + reason), then terminates the process — guaranteed even if called from inside a framework-guarded callback (an intentional stop is NOT isolated like an accidental fault).

permanent server shutdown
Used for: a clean hard stop when it is INTENDED (e.g. corrupt data, refusing to run). Pass api->plugin_handle so the stop is attributed to you. A native plugin can already crash on its own; panic makes that crash CLEAN and ATTRIBUTED.

Targeted messages — api>server2 (v12)

The targeted counterpart of server.broadcast_message: show a free-text chat line to ONE connected player (or a list of them, e.g. a guild's online members) instead of the whole server. Delivered through the engine's own replicated chat path; the engine function's signature is re-verified by reflection before every call and refused on any mismatch.

server2.send_message_to_player

int send_message_to_player(const unsigned char player_uid16[16], int channel, const char* utf8_text);

Show one chat line to the player whose raw 16-byte PlayerUId (as read from PalPlayerState.PlayerUId) matches. Returns 1 when delivered, 0 when the player is not connected or a safety gate refused. Only channel 0 (the default chat channel, proven to display in-game) is accepted today; any other value is refused until its routing is certified live. UTF-8, at most 512 UTF-16 units once decoded; malformed or over-long text is refused, never truncated.

game thread required
Used for: per-player feedback that must not spam the whole server (e.g. "your attack was blocked", "your base is under raid").

server2.send_message_to_players

int send_message_to_players(const unsigned char* player_uids16, int count, int channel, const char* utf8_text);

Same delivery for a packed array of count 16-byte PlayerUIds (count 1..128). Offline entries are skipped, not errors. Returns how many players were actually delivered to (0..count). Cost is bounded: all recipients share one object-graph lookup.

game thread required
Used for: guild-scoped notices — pass the guild's online members in one call.
In short. A PalApi plugin can: read game state, call its functions, write values safely and reversibly, observe everything that happens (ProcessEvent), offer chat and RCON commands, schedule timers, read its config, list players, broadcast or target messages, install low-level hooks, load its own dependencies, and deliberately stop the server when it needs to. All while self-adapting to game updates and cleaning up on unload. The core discipline: respect the game thread and never block.
